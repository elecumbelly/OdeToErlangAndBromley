# Tech-Debt Burn-Down — 2026-05-17

App lives in `odetoerlang/`. Package manager: `pnpm@10.33.4`. Validation gate for every PR:

```
pnpm install
pnpm exec tsc -b --force
pnpm test --run            # 569 tests
pnpm build                 # tsc -b && vite build
pnpm audit                 # confirm vuln count drops
```

CI (`.github/workflows/ci.yml`) runs `pnpm run lint`, `pnpm run build`, `pnpm run test:run`. Husky pre-commit runs `lint-staged` (eslint --fix + `vitest related`). Branch off `main`; one PR per major bump.

Cooldown hook: any package published <3 days ago is blocked. Where a target version is fresh, the plan flags it and recommends pinning to the latest version >= 3 days old at PR time.

---

## Sequencing rationale

Dependency edges that force ordering:

1. **`vite` major → `@vitejs/plugin-react` major → `vite-plugin-pwa` major.** vite-plugin-pwa 1.x peers vite 5/6/7; plugin-react 6 peers vite 7. To clear the workbox vuln chain (the single biggest audit win), vite must move first. Pair plugin-react with vite in the same PR (its peer changes track vite).
2. **`vitest` + `@vitest/coverage-v8` together.** They share the same major. vitest 4 peers vite 7, so this must come after the vite bump.
3. **`react` 19 + `@types/react` + `@types/react-dom` together.** Types lag a release behind sometimes; pin all three to the same React minor in one PR. Independent of vite chain.
4. **`recharts` 3 after react 19.** recharts 3 supports react 19 cleanly; doing recharts first would force a temporary peer-deps workaround.
5. **`eslint` 9 stand-alone.** `@eslint/js` and `typescript-eslint` are already on the 9.x line in devDependencies, so the runtime jump is mostly the binary + flat-config recipe (already flat).
6. **`lint-staged` 17 stand-alone.** Trivial. Verify Node engine still satisfied.
7. **`tailwindcss` 4 stand-alone.** Config format change (`@theme` in CSS). Large diff to `index.css` and likely removal of `tailwind.config.js` (or kept as JS-side shim). Do this last among tooling because it has the most surface to disturb.

**Recommended order** (PR numbers correspond to per-PR sections below):

1. PR1: vite-plugin-pwa 1.x (no vite bump — verify it works on vite 5 first; the package supports vite ^5||^6||^7). Quickest single audit win.
2. PR2: vite 7 + @vitejs/plugin-react 6 (paired).
3. PR3: vitest + @vitest/coverage-v8 4.x (paired).
4. PR4: eslint 9.x.
5. PR5: lint-staged 17.x.
6. PR6: react/react-dom 19 + @types/react + @types/react-dom (paired).
7. PR7: recharts 3.x.
8. PR8: tailwindcss 4.x.
9. PR9: Playwright E2E smoke tests.
10. PR10: Mobile QA sign-off (no code change — verification log + TODO close).

Total: **10 PRs**.

### PR8 — `chore(deps): bump tailwindcss to 4.x`

**Why last among tooling.** Config format change (`@theme` directive in CSS) plus PostCSS plugin rename (`@tailwindcss/postcss`). Largest diff surface in tooling.

**Changes.**
- `odetoerlang/package.json`: `tailwindcss` `^3.4.18` → `^4.x`. Add `@tailwindcss/postcss` `^4.x`. Remove `autoprefixer` if Tailwind 4 vendors it (Tailwind 4 uses Lightning CSS and handles prefixing) — verify before removal.
- `odetoerlang/postcss.config.js`: replace `tailwindcss: {}` with `'@tailwindcss/postcss': {}`. Drop `autoprefixer` only if confirmed redundant.
- `odetoerlang/src/index.css`: replace `@tailwind base;` `@tailwind components;` `@tailwind utilities;` with the single `@import "tailwindcss";` directive. Move the theme-token block (currently in `tailwind.config.js` under `theme.extend`) into a `@theme` block in CSS.
- `odetoerlang/tailwind.config.js`: in Tailwind 4 the JS config is optional. Two choices:
  - **Recommended:** Migrate token definitions (colors, fontFamily, fontSize, borderRadius, boxShadow, transitions, animation, keyframes, backgroundImage) into `@theme` blocks in `index.css`. Delete `tailwind.config.js` afterward. This matches Tailwind 4's CSS-first stance.
  - **Fallback:** Keep `tailwind.config.js` and reference it from `index.css` via `@config "../tailwind.config.js";`. Less idiomatic but reduces the migration diff. Acceptable as a transitional step if the recommended path lands too much surface for one PR.
- `html.dark` / `html.light` selectors are CSS-driven (CSS variable swaps in `index.css`), not Tailwind's `darkMode` config — no `darkMode` directive change needed.

**Known breaks.**
- `@apply` count is zero (grep verified) — no migration needed there.
- Custom shadows / colors expressed via CSS variables continue to work; the `bg-cyan` etc. utilities are generated from the `colors.cyan` token in the JS config or `@theme` block — preserve those names.
- The "legacy support" entries (`borderRadius.apple`, `boxShadow.apple`, `boxShadow.apple-lg`) — keep or audit. Don't refactor in this PR.
- Tailwind 4 changed the `gray` palette and removed some defaults. Verify the app still renders. Custom palette uses CSS variables — should be unaffected.

**Validation.**
- `pnpm build` — must produce a CSS bundle.
- `pnpm preview` — full visual smoke of dark and light theme. Tour the calculator, charts, dashboard, calendar, dialogs, sticky KPI bar.
- `pnpm test --run`.
- `pnpm audit` — expect the 2 tailwind advisories cleared.

**Effort.** L. Largest config rewrite of the set. Plan for a longer review.

**Cooldown risk.** Latest tailwind 4.x patches sometimes ship close to today. Pin a patch >= 3 days old.

**Install commands.**

**Expected runtime.** 3 specs, single browser, ~30-60s locally; ~90-120s in CI cold.

**Validation.**
